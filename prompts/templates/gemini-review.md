# Gemini Output Review Template

## Purpose

Template for reviewing code generated by Gemini during `/do-by-gemini` execution.
Used by Claude Code to analyze Gemini-generated changes and provide actionable feedback.

## Review Context

```yaml
plan_document: {plan_path}
output_log: {log_path}
execution_model: {model}
changed_files: {file_list}
```

## Review Checklist

### 1. Plan Compliance

- [ ] All required files were created/modified as specified
- [ ] Implementation matches the plan's step-by-step requirements
- [ ] No features added beyond the plan scope
- [ ] All acceptance criteria can be verified

### 2. Code Correctness

- [ ] Logic matches the expected behavior from source document
- [ ] Edge cases are handled appropriately
- [ ] No obvious bugs or errors in implementation
- [ ] Return values and data types are correct

### 3. Security Review

- [ ] No hardcoded credentials or secrets
- [ ] Input validation is present where needed
- [ ] No SQL injection vulnerabilities
- [ ] No XSS vulnerabilities in frontend code
- [ ] Proper authentication/authorization checks
- [ ] Sensitive data is handled securely

### 4. Error Handling

- [ ] Appropriate try-catch blocks where needed
- [ ] Error messages are informative but not leaking internals
- [ ] Failed operations have proper cleanup
- [ ] Network/IO errors are handled gracefully

### 5. Code Style

- [ ] Follows project coding conventions
- [ ] Consistent naming conventions
- [ ] Appropriate use of comments for complex logic
- [ ] No dead code or unused imports
- [ ] Proper indentation and formatting

### 6. Test Coverage

- [ ] Unit tests cover new functionality
- [ ] Tests verify edge cases
- [ ] Tests are properly structured
- [ ] No tests with hardcoded or flaky values

### 7. Performance Considerations

- [ ] No obvious N+1 query issues
- [ ] Large data sets are handled efficiently
- [ ] No unnecessary computations in loops
- [ ] Proper use of caching where applicable

## Review Result Structure

```yaml
summary: "Brief overall assessment"
pass_count: 0
warning_count: 0
issue_count: 0
items:
  - severity: pass|warning|issue
    category: security|performance|style|logic|test|compliance
    message: "Description of the finding"
    location: "file_path:line_number"  # optional
    suggestion: "How to fix"           # optional for issues/warnings
```

## Severity Guidelines

### Pass
- Meets or exceeds requirements
- No concerns noted
- Example: "Authentication flow implemented correctly with proper token handling"

### Warning
- Minor concerns that should be addressed
- Not blocking but recommended to fix
- Example: "Consider adding input validation for email format"
- Example: "Missing error boundary in React component"

### Issue
- Significant problems that should be fixed
- Security vulnerabilities
- Logic errors that could cause bugs
- Example: "Password stored in plain text without hashing"
- Example: "SQL injection vulnerability in user query"

## Improvement Suggestion Format

For each warning or issue, provide actionable suggestions:

```markdown
**Issue:** {brief description}
**Location:** {file}:{line}
**Category:** {category}

**Current:**
```{lang}
{problematic code}
```

**Suggested:**
```{lang}
{improved code}
```

**Rationale:** {why this change improves the code}
```

## Categories Reference

| Category | Description | Examples |
|----------|-------------|----------|
| security | Security vulnerabilities | XSS, SQL injection, auth bypass |
| performance | Performance issues | N+1 queries, memory leaks |
| style | Code style violations | Naming, formatting, dead code |
| logic | Logic errors or bugs | Wrong conditionals, off-by-one |
| test | Test quality issues | Missing tests, flaky tests |
| compliance | Plan/spec compliance | Missing features, extra features |
